# AI Editing Instructions

이 저장소를 수정하는 AI는 아래 지침을 작업 규칙으로 따른다. 코딩 실력을 과신하지 말고, 작은 변경도 의도와 회귀 가능성을 꼼꼼히 확인한다.

## 기본 원칙

- 변경 전 현재 파일 구조와 관련 코드를 먼저 읽고, 기존 패턴을 따른다.
- 사용자가 명시하지 않은 리팩터링, 디자인 변경, 파일 정리는 하지 않는다.
- 사용자 또는 다른 작업자가 만든 변경을 되돌리지 않는다.
- 큰 변경은 작게 나누어 진행하고, 각 단계마다 동작을 확인한다.
- 보안, 저장소 격리, 백업/복원, 모바일 UI, XSS 방어는 특히 보수적으로 다룬다.

## 작업 흐름

1. 먼저 `test/` 폴더의 격리된 테스트 앱에서 변경을 구현한다.
2. 테스트 앱에서 의도한 동작과 회귀 여부를 확인한다.
3. 사용자가 지시한 경우에만 검증된 변경을 본 파일에 적용한다.
4. 본 파일에 적용할 때는 테스트 앱과 운영 파일의 차이를 다시 읽고, 단순 복사로 깨질 부분이 없는지 확인한다.
5. 작업 후 전체 테스트를 실행하고 실패가 있으면 원인을 수정한다.

## 파일 범위

- 운영 파일: `index.html`, `script.js`, `style.css`, `launcher.html`
- 테스트 파일: `test/**`
- 테스트에서 먼저 검증한 뒤 운영 파일에 반영하는 것을 기본으로 한다.
- 테스트 파일만 수정한 상태에서 테스트가 정상 통과하면, 별도 확인 없이 커밋하고 푸시한다.

## 테스트

기본 회귀 테스트는 다음 명령으로 실행한다.

```bash
npx @playwright/test test test/smoke.spec.cjs
```

테스트를 실행할 수 없는 환경이면 그 이유를 명확히 기록하고, 가능한 대체 검증을 수행한다. 브라우저 설치가 필요한 경우 Playwright 안내에 따라 필요한 브라우저를 설치한 뒤 다시 실행한다.

## 에이전트 활용

효율적이라고 판단되면 보조 에이전트를 만든다. 단, 역할은 작고 명확하게 나눈다.

- 구현 에이전트: 특정 파일 또는 기능 범위만 맡긴다.
- 검증 에이전트: 변경된 동작, 테스트 누락, 회귀 위험을 따로 검토한다.
- 감독관 에이전트: 최종 검수만 맡기며, 완벽한 패치를 목표로 사소한 문제까지 지적하게 한다.

에이전트를 사용할 때도 최종 책임은 현재 작업 AI에게 있다. 에이전트 결과를 그대로 믿지 말고 직접 확인한다.

## 최종 감독관 검수

최종 제출 전 까다로운 감독관 AI 관점으로 다음을 검사한다.

- 사용자의 원래 요청을 빠뜨리지 않았는가?
- 변경 범위가 과하지 않은가?
- `test/`에서 검증한 내용이 운영 파일에 안전하게 반영될 수 있는가?
- 테스트가 통과했는가?
- 실패 또는 미실행 테스트가 최종 보고에 솔직히 적혔는가?
- 보안, 저장소 분리, 데이터 보존, 모바일 동작에 회귀가 없는가?
- 커밋/푸시 조건을 정확히 지켰는가?

감독관 검수에서 문제가 나오면 수정 후 다시 검수한다. 최종 검수 결과가 `"합격"`이 될 때까지 반복한다.

## 커밋과 푸시

- `test/` 파일을 수정했고 테스트가 정상 통과한 경우, 묻지 말고 커밋 후 푸시한다.
- 운영 파일까지 수정한 경우에도 사용자의 지시와 테스트 결과를 확인한 뒤 커밋 범위를 신중히 잡는다.
- 커밋 전 `git status --short`로 변경 파일을 확인한다.
- 커밋 메시지는 변경 목적과 검증 내용을 짧고 명확하게 쓴다.
